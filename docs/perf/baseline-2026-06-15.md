# 效能基準 / Performance Baseline — Catalog Browse (2026-06-15)

First load-test baseline for the project (PLAN.md §13 Phase 1 deliverable: *"First perf baseline: k6 catalog browse run, recorded in `docs/perf/baseline-{date}.md`"*). It establishes the storefront read path's latency + throughput SLOs so later optimization work (Phase 10) and the nightly `load-test.yml` job have something to compare against.

Script: [`tests/load/catalog-browse.js`](../../tests/load/catalog-browse.js) · Tool: **k6 v2.0.0**

## Scenario

Models an anonymous shopper browsing the public catalog. Each iteration walks the hot read path with realistic think-time between steps:

| Step | Request | Think time |
|---|---|---|
| 1. Filter panel | `GET /api/v1/catalog/categories` | 0.5s |
| 2. Product grid | `GET /api/v1/catalog/products?Page=1&PageSize=12` (50% with a random `&CategoryId=` filter) | 1.0s |
| 3. Product detail | `GET /api/v1/catalog/products/{slug}` (random live slug) | 1.0s |

`setup()` discovers live slugs + category ids first, so the script isn't pinned to a seed set. Each response is asserted (status 200 + expected body shape); a failed assertion counts against a custom `browse_errors` rate.

**Load profile:** ramp 0→20 VUs over 30s, hold 20 VUs for 1m, ramp down over 30s (≈2 min total).

## Environment

> ⚠️ **Dev-box baseline, not production.** Read the numbers as a *relative* floor for regression detection, not an absolute SLA.

| Aspect | Value |
|---|---|
| API | ASP.NET Core 10, **Debug** build via `dotnet run`, single instance on `http://localhost:5124` |
| Datastore | SQL Server (local dev instance, `RetailOms`) |
| Host | Linux 7.0.0, 20 vCPU, 29 GiB RAM, .NET 10.0.109 |
| Network | loopback (no real network latency) |
| **Dataset** | **3 products, 1 category** (the seeded demo set) — small; see caveats |
| Run date | 2026-06-15 |

## Thresholds (recorded SLOs)

These are encoded in the script's `thresholds` and gate future runs:

- `http_req_duration` **p(95) < 500ms**
- `http_req_failed` **rate < 1%**
- `checks` **rate > 99%**
- `browse_errors` **rate < 1%**

## Results — ✅ all thresholds passed

| Metric | Value |
|---|---|
| Iterations | 732 (6.06/s) |
| HTTP requests | 2,198 (18.2 req/s) |
| **Failed requests** | **0.00%** (0 / 2,198) |
| **Checks passed** | **100.00%** (3,660 / 3,660) |
| Peak VUs | 20 |
| Data received | 1.6 MB |

**Latency — `http_req_duration`:**

| avg | min | med | p(90) | p(95) | p(99) | max |
|---|---|---|---|---|---|---|
| 3.75ms | 0.96ms | 3.58ms | 5.08ms | **5.67ms** | 7.13ms | 38.69ms |

## Interpretation

- The catalog read path is **healthy under 20 concurrent VUs** — p95 of 5.67ms is ~88× under the 500ms budget, zero errors, flat latency distribution (p99 7ms, max 39ms). No connection-pool exhaustion or tail blow-up at this concurrency.
- Throughput (~18 req/s) is **think-time-bound, not server-bound** — with 2.5s of `sleep()` per iteration, 20 VUs can only offer ~8 iterations/s. This run measures **latency under light concurrency**, not max throughput. A saturation/stress run (no think time, arrival-rate executor) is the natural follow-up.

## Caveats / known limitations

1. **Tiny dataset (3 products).** Queries return trivially small result sets and SQL Server caches everything, so absolute latencies are optimistic. A representative baseline needs a fuller catalog (≥ a few hundred products across several categories) to exercise paging, filtered scans, and larger payloads. **Re-run after seeding a realistic catalog.**
2. **Debug build + loopback.** A `Release` build behind the container/APIM topology (real serialization + network hops) will be slower. PLAN.md §13 targets a nightly k6 run against *staging* — that's the apples-to-apples number for production SLAs.
3. **Latency-only profile.** This is not a stress test; it doesn't find the saturation point. Add a high-arrival-rate scenario when chasing the Phase 10 optimization targets.
